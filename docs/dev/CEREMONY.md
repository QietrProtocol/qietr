# Trusted-Setup Ceremony — runbook & tooling

> **Goal:** replace the single-contributor **dev verifying key** (`qietr-pool/.../dev_vk.rs`,
> labeled "DEV KEY. Do not deploy.") with a verifying key produced by a
> **multi-party** Groth16 Phase-2 ceremony, so no single party knows the
> toxic-waste trapdoor that would let them forge proofs and drain the vaults.
>
> This document is the operational runbook. The conceptual phase background is in
> [`11-trusted-setup.md`](11-trusted-setup.md). The multi-party run itself is a
> **human/external** task — this repo can produce the tooling and verify the
> output, but the contributors and their entropy are real people.

## Why it matters

The on-chain verifier (`verifier.rs`) hardcodes `VERIFYINGKEY`. With a dev key
generated by one machine, whoever held that machine's randomness can forge any
proof. The `withdraw` instruction now enforces that the embedded VK matches the
timelocked `PoolConfig.verifying_key_hash` (so the VK can't be silently swapped),
but the *key itself* is only as trustworthy as its setup. A real ceremony makes
the trapdoor unknown unless **all** contributors collude.

## Circuit under ceremony

- Circuit: `qietr_payment` (`qietr-circuits/circuits/qietr_payment.circom`)
- Constraints: ~5,789 non-linear → fits `pot14` (2^14).
- Curve / proof system: BN254 / Groth16.
- Public-signal order (authoritative): `[amount, root, nullifierHash, recipient,
  paymentAmount, changeCommitment]`.

## Phase 1 — Powers of Tau (perpetual / universal)

Use an existing, well-attested universal Powers-of-Tau (e.g. Hermez `pot14`)
rather than running a fresh Phase 1. If running fresh, ≥3 independent
contributors, each on an air-gapped machine, beacon-finalized.

Verify whatever `.ptau` you adopt:

```bash
snarkjs powersoftau verify pot14_final.ptau
```

## Phase 2 — circuit-specific

```bash
# 0. Compile + export the r1cs (committed under qietr-circuits/build).
#    (See qietr-circuits/README.md "Quick start".)

# 1. Initialize phase 2 from the universal ptau.
snarkjs groth16 setup qietr_payment.r1cs pot14_final.ptau qietr_0000.zkey

# 2. Each contributor (N >= 3, independent, ideally on air-gapped hardware):
snarkjs zkey contribute qietr_<i-1>.zkey qietr_<i>.zkey \
  --name="contributor-<i>" -e="$(head -c 64 /dev/urandom | base64)"
#   - publish the resulting transcript hash so contributions are auditable
#   - delete local entropy after contributing

# 3. Apply a public, verifiable random beacon to finalize.
snarkjs zkey beacon qietr_<N>.zkey qietr_final.zkey \
  <beacon-hash> 10 -n="final beacon"

# 4. Export the verification key + verify the full transcript.
snarkjs zkey verify qietr_payment.r1cs pot14_final.ptau qietr_final.zkey
snarkjs zkey export verificationkey qietr_final.zkey verification_key.json
```

## Converting the VK for the on-chain program

The program embeds the VK as Rust (`dev_vk.rs` → replace with `prod_vk.rs`). The
conversion must preserve the BN254 byte layout `groth16-solana` expects (G1 64
bytes, G2 128 bytes in `x1||x0||y1||y0` order; IC entries concatenated). A
converter script lives at `qietr-circuits/scripts/` (`vk-to-rust`); if absent,
write one that maps `verification_key.json` → the `Groth16Verifyingkey` struct,
matching the packing in `qietr-sdk/src/prover.ts` (`packG2`).

After generating `prod_vk.rs`:

1. `cargo test -p qietr_pool --lib poseidon_parity` (sanity).
2. Re-run the SDK ↔ on-chain e2e (`qietr-pool/scripts/devnet-e2e.mts`) against a
   locally redeployed program using the new VK.
3. Compute the new `embedded_vk_hash()` and roll it through the timelock:
   `queue_vk_upgrade(new_hash)` → wait `PENDING_VK_LOCK_SLOTS` (~48h) →
   `apply_vk_upgrade`, deploying the new program at apply time. Because
   `withdraw` enforces `embedded_vk_hash() == config.verifying_key_hash`,
   sequence the redeploy with the apply so withdraws aren't interrupted.

## Publishing for auditability

- Publish every contribution transcript hash and the final `verification_key.json`.
- Publish the beacon source and value.
- Publish `prod_vk.rs` alongside the `snarkjs zkey verify` output so anyone can
  reproduce the VK → Rust conversion.

## Checklist

- [ ] ≥3 independent Phase-2 contributors, entropy destroyed
- [ ] Public beacon finalization
- [ ] `snarkjs zkey verify` passes on the final zkey

- [ ] VK hash rolled through the 48h timelock with a coordinated redeploy
- [ ] `dev_vk.rs` removed from the build; `verifier.rs` points at `prod_vk.rs`
- [ ] Transcripts + final VK published
